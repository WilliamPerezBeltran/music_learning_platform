# CONTEXT.md вҖ” Music Learning Platform

## Vision
Educational platform for learning piano through a color-based visual system.
Children and beginners can play songs quickly without prior music knowledge.

Core combination:
- Colored sheet music
- Interactive playback
- Progressive levels per song
- Guided tutorials
- Freemium model

## Business Model
- Free: first songs, first levels, basic features
- Premium: full library, all levels, weekly new songs, progress tracking, PDF export

## MVP Validation Hypothesis
A user with zero music knowledge can:
1. Choose a song
2. Follow the color-coded score
3. Listen to the playback
4. Learn a basic melody
5. Progress through levels faster than with traditional notation

## Phase Roadmap (high level)
- Phase 0: Technical validation вҖ” prove the stack works
- Phase 1: MVP Core вҖ” library, colored score, playback, visual config
- Phase 2: Level system вҖ” 5 difficulty levels per song
- Phase 3: Content admin вҖ” add songs without code changes
- Phase 4: Initial catalog вҖ” 20вҖ“50 songs, beta launch
- Phase 5: Landing page вҖ” acquisition
- Phase 6: User accounts вҖ” persist progress
- Phase 7: Progress tracking вҖ” metrics per song/level
- Phase 8: Monetization вҖ” first paying users
- Phase 9: Social media content вҖ” organic acquisition funnel
- Phase 10: Advanced features вҖ” virtual piano, MIDI input, evaluation
- Phase 11: AI вҖ” audioвҶ’MusicXML, auto level generation (only after market validation)

## Level System (per song)
```
Level 1 вҖ” Main melody only
Level 2 вҖ” Simplified melody
Level 3 вҖ” Melody + basic chords
Level 4 вҖ” Simplified left hand
Level 5 вҖ” Full song
```

## Initial Song Catalog (Phase 0 test songs)
- Bartolito (primary demo)
- Estrellita
- Feliz CumpleaГұos
- Los Pollitos
- Himno de la AlegrГӯa

Each song needs: MusicXML file, lyrics, note colors, available levels.

## Visual Configuration (user-controlled toggles)
- Colors on/off
- Note names (Do, Re, Mi...) on/off
- Chords on/off
- Left hand on/off
- Right hand on/off
- Lyrics on/off

## Data Model
See `docs/technical/phase_0/01_Core/entities.md` for full field definitions.

Relationships:
```

 в””в”Җв”Җ SongVersion (1вҶ’N)   вҖ” one version per difficulty level
      в””в”Җв”Җ MusicTimeline (1вҶ’1)  вҖ” processed temporal data
           в””в”Җв”Җ MusicalEvent (1вҶ’N)  вҖ” individual note/rest events

See `docs/technical/phase_0/01_Core/architecture.md` for full diagram.

## MusicXML File Storage
Decision pending. Options evaluated:
- `priv/static/` вҖ” simplest for Phase 0
- S3 / Cloudflare R2 вҖ” for production scale
- Git repository вҖ” for versioning during development

Recommended for Phase 0: `priv/static/songs/`

## Color Mapping System
One color per musical note (DoвҶ’ReвҶ’Mi... mapped to distinct colors).
Color applied to: note head in SVG, note name label, highlight overlay during playback.
Mapping defined in `color_mapper.ex` (Visual layer).

## Synchronization Strategy
Highest technical risk of Phase 0.

The sync problem: MusicXML defines notes structurally, Tone.js plays in real time,
OSMD renders statically. The three must stay aligned during playback.

Solution architecture:
- MusicXML is parsed once into an Event Timeline (list of timed events)
- Tone.js Transport fires callbacks at each event's start_time
- Each callback triggers: audio note play + OSMD highlight update
- Seeking/pause handled by resetting Transport position against the timeline

All sync goes through the Sync Engine вҖ” OSMD and Tone.js never communicate directly.

## JS Hooks Architecture
LiveView cannot call JavaScript directly. The bridge:
```
LiveView
  вҶ“ pushEvent / handle_event
JS Hook (osmd_hook.js | tone_player_hook.js | sync_hook.js)
  вҶ“
OSMD / Tone.js
```

Hooks are mounted on DOM elements via `phx-hook` attribute.
They receive events from LiveView and translate them to JS engine calls.

# Arquitectura TГ©cnica Recomendada (Mejorada)

## 1. Backend (Core System)

### Phoenix + Elixir

Responsable de:

- OrquestaciГіn del estado de la aplicaciГіn
- GestiГіn de canciones, niveles y configuraciГіn
- CoordinaciГіn de sincronizaciГіn (tiempo musical lГіgico)
- APIs internas para LiveView

### Concepto clave:
El backend **NO reproduce audio ni renderiza mГәsica**, solo coordina.

---

### Background Jobs / Workers (Capa AsГӯncrona)

Responsable de:

- Procesamiento de MusicXML
- GeneraciГіn de niveles de canciones
- ValidaciГіn y normalizaciГіn de archivos musicales
- Pre-cГЎlculo de estructuras para el Sync Engine
- Tareas pesadas fuera del flujo de LiveView

### Concepto clave:
Los workers son el backend diferido, no afectan la interacciГіn en tiempo real.

---

### Event Timeline Layer (NГҡCLEO MUSICAL)

Responsable de:

- Convertir MusicXML en eventos temporales
- Representar la canciГіn como una lГӯnea de tiempo
- Servir como fuente de verdad del sistema musical
- Soportar niveles de dificultad (0.1 вҶ’ 0.10)

### Concepto clave:
 Todo se sincroniza contra la timeline, no contra OSMD ni Tone.js

---

### State Model (CONTROL GLOBAL)

Responsable de:

- Estado de reproducciГіn (play/pause/stop)
- Tiempo actual de la canciГіn
- Nivel activo
- CanciГіn activa
- ConfiguraciГіn visual global

### Concepto clave:
 Es la Гәnica fuente de verdad del sistema en runtime

---

## 2. Frontend (UI Layer)

### Phoenix LiveView

Responsable de:

- UI reactiva en tiempo real
- Estado de reproducciГіn visual
- InteracciГіn del usuario
- ComunicaciГіn con JS hooks

### Principio:
LiveView es el вҖңcerebro de UIвҖқ, no el motor musical.

---

## 3. JS Bridge Layer (COMUNICACIГ“N)

Responsable de:

- Conectar LiveView вҶ” JavaScript
- Emitir eventos de reproducciГіn
- Sincronizar estado con Tone.js y OSMD
- Manejar hooks del frontend

### Componentes:

- OSMD Hook
- Tone.js Player Hook
- Sync Controller Hook

### Concepto clave:
 Es el puente obligatorio entre backend y motor musical

---

## 4. Audio Engine

### Tone.js (JavaScript)

Responsable de:

- ReproducciГіn de audio en tiempo preciso
- Scheduler musical (Transport)
- Control de tempo (BPM / speed)
- Timing de notas

### Regla crГӯtica:
Tone.js es la **Гәnica fuente de audio**

---

## 5. Rendering Engine (Partituras)

### OpenSheetMusicDisplay (OSMD)

Responsable de:

- Renderizar MusicXML como SVG
- Mostrar pentagramas y notas
- Highlight de notas activas

### IntegraciГіn:
Se conecta vГӯa JS Bridge Layer

---

## 6. Data Layer (Formato Musical)

### Formatos soportados

- MusicXML (principal)
- MIDI (futuro)

### Responsabilidades

- Representar canciones
- Definir niveles de dificultad
- Convertir datos вҶ’ Event Timeline

---

## 7. Sync Engine (CRГҚTICO)

Responsable de:

- Sincronizar Timeline вҶ’ Audio + Visual
- Coordinar OSMD y Tone.js
- Mantener consistencia temporal

### Modelo mental:

```text
MusicXML вҶ’ Event Timeline вҶ’ Sync Engine вҶ’ (Audio + Visual)
```

# ExplicaciГіn simple del flujo

## 1. Usuario interactГәa

- Da Play  
- Cambia velocidad  
- Selecciona canciГіn  

---

## 2. LiveView (UI Layer)

- Recibe acciГіn  
- Actualiza estado  
- EnvГӯa evento a JS Bridge  

---

## 3. JS Bridge Layer

- Conecta backend вҶ” frontend  
- EnvГӯa comandos a:  
  - OSMD (visual)  
  - Tone.js (audio)  

---

## 4. OSMD + Tone.js

- OSMD вҶ’ dibuja partituras  
- Tone.js вҶ’ reproduce audio  

---

## 5. Backend (cerebro real)

### State Model
- controla TODO el estado global  

### Event Timeline
- convierte mГәsica en eventos con tiempo real  

### Sync Engine
- asegura que audio + visual van sincronizados  

---

## 6. Workers (background)

- preparan canciones antes de reproducirse  
- generan niveles  
- procesan archivos pesados  


### IDEA CLAVE DEL SISTEMA
```text
MusicXML
   вҶ“
Event Timeline (verdad musical)
   вҶ“
Sync Engine (coordinador)
   вҶ“
Audio (Tone.js) + Visual (OSMD)
```

### RESUMEN ULTRA SIMPLE

- LiveView = UI
- JS Bridge = conexiГіn
- Tone.js = sonido
- OSMD = visual
- Event Timeline = mГәsica real
- Sync Engine = cerebro del sistema
- Workers = procesamiento en background
- State Model = control global

# CLAUDE.md вҖ” Music Learning Platform

## Project
Web platform for learning piano using a color-coded sheet music system.
Target users: children and beginners with zero music theory.
Core value: colored notation + progressive levels per song.

## Tech Stack
- Backend: Phoenix + Elixir
- Frontend: Phoenix LiveView
- Audio engine: Tone.js (JavaScript)
- Score renderer: OpenSheetMusicDisplay (OSMD)
- Music format: MusicXML (primary), MIDI (future)
- Database: PostgreSQL
- CSS: Tailwind CSS

## Architecture вҖ” Mental Model
```
MusicXML вҶ’ Workers вҶ’ Event Timeline вҶ’ Sync Engine вҶ’ State Model
                                                          вҶ“
                                                      LiveView
                                                          вҶ“
                                                      JS Bridge
                                                     вҶҷ        вҶҳ
                                                 OSMD       Tone.js
```

Backend does NOT play audio or render music вҖ” it only coordinates.

## Layer Responsibilities
- State Model: runtime source of truth (active song, level, play/pause, tempo)
- Event Timeline: MusicXML вҶ’ timed events вҖ” the musical source of truth
- Sync Engine: keeps audio + visual in sync against the timeline
- JS Bridge (hooks): LiveView вҶ” OSMD/Tone.js communication
- Workers: async MusicXML processing, level generation, normalization
- OSMD: SVG score rendering + note highlight
- Tone.js: audio playback, Transport scheduler, BPM control

## Code Structure
```
lib/
в”ңв”Җв”Җ music_learning/
в”Ӯ   в”ңв”Җв”Җ state/       # state_model.ex, playback_state.ex, session_state.ex
в”Ӯ   в”ңв”Җв”Җ sync/        # sync_engine.ex, time_coordinator.ex, note_tracker.ex
в”Ӯ   в”ңв”Җв”Җ timeline/    # event_timeline.ex, musicxml_parser.ex, timeline_builder.ex
в”Ӯ   в”ңв”Җв”Җ songs/       # song.ex, level.ex, song_library.ex, song_loader.ex
в”Ӯ   в”ңв”Җв”Җ workers/     # musicxml_worker.ex, level_generator_worker.ex
в”Ӯ   в”ңв”Җв”Җ playback/    # playback_controller.ex, tempo_controller.ex, audio_sync.ex
в”Ӯ   в””в”Җв”Җ visual/      # color_mapper.ex, notation_config.ex, highlight_engine.ex
в””в”Җв”Җ music_learning_web/
    в”ңв”Җв”Җ live/song_live/  # show.ex, controls_component.ex, score_component.ex
    в”ңв”Җв”Җ hooks/           # osmd_hook.js, tone_player_hook.js, sync_hook.js
    в””в”Җв”Җ router.ex
```

## Core Entities
- Song: base entity (title, category, duration)
- SongVersion: pedagogical level (melody | simplified | chords | full)
- MusicTimeline: processed temporal structure (bpm, total_duration, resolution)
- MusicalEvent: minimum sync unit (pitch, start_time, end_time, voice, color_key)
- PlaybackSession: real-time playback state (current_time, is_playing, speed)
- ContentAsset: files associated to a song (musicxml | audio | image | midi)

## Phase 0 вҖ” Technical Validation (current)
Goal: prove the stack can support the product.

- 0.1 Phoenix project + Tailwind + test page
- 0.2 Integrate OSMD вҖ” render static MusicXML
- 0.3 Load MusicXML from files вҖ” switch between songs
- 0.4 Integrate Tone.js вҖ” play notes
- 0.5 Sync score + audio  вҶҗ highest risk
- 0.6 Highlight current note
- 0.7 Play / Pause / Stop controls
- 0.8 Speed control
- 0.9 Color system per note
- 0.10 Functional demo with "Bartolito"

## Critical Rules
- Musical timing logic belongs in Event Timeline, never in LiveView
- Tone.js is the only audio source вҖ” no other audio APIs
- All sync goes through Sync Engine вҖ” OSMD never talks directly to Tone.js
- JS Bridge (hooks) is the mandatory bridge between LiveView and JS engines
- Workers handle heavy processing вҖ” never inside mount or handle_event
